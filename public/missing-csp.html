CSP・セキュリティヘッダーの不備 VULNERABLE

このページ（およびサイト全体）には Content-Security-Policy が設定されていません。 そのため、インラインスクリプトや外部スクリプトが無条件に実行できてしまいます。

下のボタンは、インラインの onclick ハンドラをそのまま実行します（CSP があればブロックされます）。

本来設定すべきセキュリティヘッダーの例

Content-Security-Policy: default-src 'self'; script-src 'self'; object-src 'none'; base-uri 'self'; frame-ancestors 'none'
X-Content-Type-Options: nosniff
X-Frame-Options: DENY
Referrer-Policy: strict-origin-when-cross-origin
Permissions-Policy: geolocation=(), microphone=(), camera=()

備考: GitLab Pages で任意の HTTP レスポンスヘッダーを付与できるかどうかは GitLab のバージョンや設定に依存します。確実に付与したい場合は、Pages の前段に CDN／リバースプロキシ（Cloudflare, CloudFront 等）を置いてヘッダーを付与する構成が安全です。 詳細は必ず最新の GitLab Pages 公式ドキュメントを確認してください（未確認の情報です）。

FIX（考え方）

  • まずは script-src 'self' のように厳格な CSP から始め、必要な範囲だけ緩める
  • インラインスクリプト（onclick="..." や <script>...</script>）は避け、外部 JS ファイル + イベントリスナーに置き換える
  • やむを得ずインラインを許可する場合は nonce / hash ベースの CSP を使う
  • CSP はあくまで多層防御の一つであり、XSS 自体を作り込まないこと（サニタイズ・エスケープ）が最優先
